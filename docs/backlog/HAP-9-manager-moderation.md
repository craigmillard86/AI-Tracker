## Story
As a Manager, I review each direct report's self-scores and evidence side by side, adopt or diverge (comment forced at Δ≥2, carry-forward default), producing the moderated score of record — and my report sees exactly what I decided and why.

### Dev attempt 1 — panel-fix batch 3 (2026-07-22, red-team BLOCKING — L3 score-oracle leak)

Re-panel red-team found a BLOCKING L3 privacy leak: a caller who is a chain reviewer-of-record but has NO individual-read capability (a HIG Executive over a Portfolio Leader, FR-025 clause 2) could MODERATE — and the FR-009 forced-comment error text leaked the subject's exact self-score as a repeatable oracle (SetManager throws pre-SaveChanges, so no audit, GET stays 404). Fixed both parts, plus a root cause I found while fixing:

- **ROOT CAUSE — endpoints discarded the caller's real grants.** `GetMemberAssessmentAsync` and `ModerateAsync` (and the queue) passed `CallerContext.Ungranted(callerPersonId)`, so the seam's role-capability gate never saw that a caller was an Exec/Group/Admin/BuDelegate — an above-BU leader with reports classified as a plain Manager. Added `ManagerModerationService.CallerContextAsync` (fresh `RoleGrant` DB read → real `CallerGrant`s, per HAP-4 A3 the BU anchor comes from the DB not the cookie); all three surfaces now build and pass the real caller context. This alone closes the member-GET leak (an Exec is now denied the read too, matching the pinned gateway behaviour).
- **PRIMARY — moderation ⊆ read.** `AssessmentReads.AuthorizeModeration` now requires BOTH `AuthorizeIndividualRead(...).Allowed` (role capability + reach + reader eligibility) AND `ReviewerOfRecord(subject) == caller`. A capability-less chain ancestor is denied; a BU delegate reads BU-wide but only moderates their own reviewee-of-record; legitimate employee managers + the DR-0006/FR-070-escalated employee ancestor are unaffected (they have read capability over their reviewee).
- **Queue consistency (defense-in-depth, same root cause — flagged for the panel).** `GetReviewQueueAsync` now returns an empty/not-a-manager queue for a caller with no individual-read capability, so the queue can't leak an aggregates-only role their direct reports' names + assessment states. (Not in the red-team's two-part ask, but the same leak class — closing it here to avoid a sibling finding.)
- **DEFENSE-IN-DEPTH — score values scrubbed from error messages.** `ManagerCommentRequiredException` no longer embeds `(self X, manager Y)` or the exact divergence; the client-visible 422 states only the dimension + the public threshold. Scores kept as structured properties (tests/diagnostics), never in the HTTP body. `ScoreOutOfRangeException` messages only ever contain the caller's OWN submitted score, so they're safe.
- **Comment nits** fixed (ManagerModerationService moderation doc + TeamEndpoints 404 comment now say "reviewer of record / read capability", not "direct manager").

Tests (TDD): gateway `Moderation_is_denied_for_a_reviewer_of_record_who_has_no_individual_read_capability` (Exec/Admin/GroupViewer grant denied moderation even as reviewer-of-record; ungranted manager still allowed — the contrast); HTTP `A_capability_less_direct_manager_cannot_read_moderate_or_see_a_queue` (ADMIN as EMPADM's direct manager: empty queue, GET 404 + no audit, PUT 404, assessment untouched); HTTP `A_divergence_comment_required_422_body_leaks_no_score_values`. Legitimate-manager + contractor-escalation cases stay green (regression guard).
