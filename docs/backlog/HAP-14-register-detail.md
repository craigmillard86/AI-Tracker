### Adversarial attempts specific to this story's own integrity/authority model

1. **Field smuggling (BU reassignment / stage-jump via raw JSON on PUT and POST /stage):** ATTEMPTED, FAILED TO BREACH. Sent raw JSON bodies with extra `businessUnitId`/`currentStage` properties the DTOs don't declare — silently discarded by the default `System.Text.Json` binder (no `UnmappedMemberHandling` configured anywhere in `Hap.Api`); BU and stage remained untouched.
2. **Same-BU-but-unauthorised equivalence classes:** ATTEMPTED, FAILED TO BREACH. Dev's own suite only covers a different-BU individual as the negative case; added three narrower same-BU rows to the decision table — plain manager without ownership, individual contributor without ownership, and a Group Leader (structurally a manager per the endpoint's own class-doc precedence note) — all 404, confirming `CanEditAsync`'s narrow owner/creator/BuLead-only branch has no leadership-anchor leak.
3. **Genuine two-request HTTP concurrency race on POST /stage:** ATTEMPTED. First version of the test asserted a fixed outcome (exactly one 200, one 409) and FAILED on first run — investigation showed this was the test's own wrong assumption, not a guard defect: the in-memory WebApplicationFactory test host processed the two requests sequentially (second request's read legitimately saw the first's committed state and made a further valid forward move), never producing genuine overlapping reads. Rewrote to assert the interleaving-independent invariant — strictly monotonic forward-only history, and OK-response-count == history-row-count — which held deterministically across 5 repeated runs plus the final verify.sh run. Combined with Dev's own deterministic tracked-scope proof that a genuinely stale write throws `DbUpdateConcurrencyException` and maps to 409, this is sufficient evidence the xmin guard holds; a fully genuine two-process race (separate OS processes, not just separate HttpClient/Task instances against one in-memory host) remains untested and is the same acknowledged limitation Dev's own doc comment already named.
4. **Negative customers-in-production via the weekly-update path:** ATTEMPTED, FAILED TO BREACH. 422, value never persists (reads back null).
5. **Append-only guard genuineness (direct DbContext mutation attempt):** ATTEMPTED, FAILED TO BREACH. See AC-by-AC section above.
