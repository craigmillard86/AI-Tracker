## Story
As the platform, every read of assessment data must pass through one authorisation layer — management-chain resolution, role scoping, and small-group suppression with complement defense — so no query path can ever reach a score the caller isn't entitled to. This is the constitution's Wave-0 spike, hardened into production code.

## Context

- Plan: research **D1** (seam enforcement: internal EF configs + architecture test + per-role integration tests) and **D2** (complement algorithm: suppress if n<4 OR 0 < parent.n − node.n < 4, evaluated over the fixed hierarchy with suppressed siblings excluded); constitution Art. VI.1, Art. VII Wave-0 spike.
- Files: `backend/src/Hap.Api/Authorization/**` (ChainResolver, RoleScope, Suppression, AssessmentReads gateway — the ONLY type touching Assessments/AssessmentScores DbSets), `backend/tests/Hap.Architecture.Tests/**`, integration tests in `Hap.Api.Tests`. **No migration and no DbSet registration** (audit 2026-07-21): define the Assessment/AssessmentScore domain *types* only — registering DbSets without their migration would fail verify's pending-model-change/idempotence check. The gateway is fully unit-tested on the org side now; HAP-8 registers the DbSets + migration and extends the architecture test to the DbSet form.
- Blocked by: HAP-3, HAP-4
- Parallelisable: no (everything reading assessment data depends on it)
- **HAP-4 hard constraints (copied here so they are not missed — §6 Phase 1 does not require reading all of QUESTIONS.md):**
  - **Q-014 hard-block (owner-ratification gate):** `HierarchyRoleResolver`'s computed leadership tiers (BU Lead/Group Leader/Portfolio Leader) are derived by manager-chain **depth from a validated root**, which assumes a uniform-depth org tree — correct for the synthetic generator but misassigns on real org shapes (an interim/dual-hat manager layer, or a missing tier). **This seam MUST NOT consume those computed tiers for visibility scope until Q-014 is owner-ratified with a structural "leads this unit" anchor, OR this story's own L3 red-team independently clears the derivation.** Build role-scope from a structural source (BU/Group/Portfolio FK membership + explicit RoleGrant), not from the depth-derived label, unless/until ratified.
  - **RoleGrant re-read (HAP-4 A3):** BU-scoped (`BuDelegate`) grants flatten to a bare role string in the auth cookie — the `BusinessUnitId` scope is NOT in the claim, and claims persist up to the 8h sliding window. This seam MUST re-read `RoleGrant` rows from the DB for any BU-scoped decision; never trust the cookie claim alone.
- **Admin-endpoint `[PA]` gate is already DONE (HAP-4):** the PlatformAdmin (403) policy now gates `/api/admin/sync`, `/api/admin/overrides`, and `/api/admin/frameworks`. The AC below is therefore scoped to the **assessment-read** role-scope matrix, not the admin endpoints.
