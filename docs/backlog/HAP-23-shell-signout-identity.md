## Story
As any signed-in user I can sign out from the app shell and I can see who I am currently signed in as — so I can end my session and (in the local dev provider) switch roles without clearing browser cookies by hand.

## Context
- **Found in owner acceptance testing 2026-07-22:** the running app has no way to log out. The plumbing already exists — `signOut()` in `app/src/api/client.ts` (`POST /auth/signout`) and the endpoint `IdentityEndpoints.MapPost("/auth/signout", …)` — but nothing in the shell calls it, and the top bar's "Signed in as …" is a hard-coded `strings.shell.rolePlaceholder`, not the real identity.
- Identity port: HAP-4 (FR-055/FR-056) shipped `POST /auth/signin` (dev role-picker) + `POST /auth/signout` + `GET /api/me` (returns the caller's `person_id` + roles). There is no separate sign-out FR — sign-out is part of the identity-port surface.
- Design: DESIGN.md A6 (fixed deep-navy top bar); the control lives in the existing `.app-user` span in `app/src/shell/AppShell.tsx`. Build from A1–A6 tokens only; no new tokens (addendum-update-first if a new component is needed). No mockup exists for the shell chrome (Q-004: admin/shell surfaces ship minimal, DESIGN.md-conformant).
- After sign-out the app must return to the dev sign-in role-picker (the unauthenticated state HAP-4 already renders).

## Acceptance criteria
- [x] The shell top bar shows the ACTUAL signed-in identity — display name + role(s) — read from `GET /api/me` (replacing the hard-coded `rolePlaceholder`). Falls back gracefully if `/api/me` is unavailable.
- [x] A sign-out control (button, keyboard-focusable, visible focus outline) in the top bar calls `signOut()` → `POST /auth/signout`, then returns the app to the sign-in role-picker (no stale authenticated view, no cached identity left in memory).
- [x] Sign-out clears the client session cleanly: a subsequent `GET /api/me` (or any `/api/**` call) is unauthenticated (401) until the user signs in again. (Server-side behaviour of the existing `POST /auth/signout`, unchanged by this story; client wiring calls it correctly — see notes.)
- [x] DESIGN.md A6 conformance: tokens only (deep-navy top bar), the control styled per the addendum; strings externalised (`en.ts`); colour never the sole signal.
- [x] vitest-axe passes on the shell with the new control; keyboard-only sign-out works.
- [x] `./scripts/verify.sh` green.

### Acceptance criteria — verified literally, clause by clause

Proved this is real, not theoretical, with a new test that holds `/api/me` pending and asserts the fallback text is on screen before resolving it (`shell.test.tsx`, "shows 'Identity unavailable' while GET /api/me is still pending…"). This is technically accurate at the instant it's shown (identity isn't available *yet*) so it does not violate the AC's literal wording, but it works against the AC's intent (loading is indistinguishable from failure) — recommend a follow-up story/ticket, not a blocker for this one.

2. **Sign-out control: keyboard-focusable, visible focus outline, calls signOut() → POST /auth/signout, returns to the picker, no stale view, no cached identity left in memory.** — PASS. Confirmed keyboard-focusable + focus-outline-not-suppressed (existing test, re-verified). Confirmed `signOut()` → `POST /auth/signout` fires (existing + new tests). **The composition claim — "returns to the sign-in role-picker" — was previously untested above the AppShell-alone level** (every prior test mocked `onSignedOut` and never asserted what App's real callback produces). Added an App-level round trip: render `App` authenticated → click "Sign out" → assert `SignInScreen` (role-picker) renders and both `strings.appName` and the sign-out button are gone (`app-gate.test.tsx`, "signing out from the shell returns the app to the sign-in role-picker…"). No cached identity: `me` is a plain `useState` local to `AppShell` (no module-level cache in `api/client.ts`), and `App` unmounts `AppShell` on sign-out (state flip to `'anonymous'`) — inspected and confirmed, consistent with the code reviewer's independent finding.

3. **AC-3 (verified hard): subsequent GET /api/me (or any /api/**) is unauthenticated (401) until re-signin.** — PASS, real server round-trip, not client wiring. `backend/tests/Hap.Api.Tests/Identity/SignInFlowTests.cs:78-92`, `Post_auth_signout_clears_the_session`: same `HttpClient` (cookie-carrying) does `GET /api/me` (200) → `POST /auth/signout` (200) → `GET /api/me` (asserted 401). This genuinely proves server-side session invalidation, not merely that the client called the right URL. Unchanged by this diff (no backend files touched). **Documented, non-blocking gap surfaced while probing this AC:** `signOut()` in `api/client.ts` never checks `response.ok`, and `AppShell`'s `.finally(onSignedOut)` runs regardless — so a non-2xx from `POST /auth/signout` would still return the caller to the picker client-side while the server-side cookie might survive. Wrote a test proving the client-visible half of this (`shell.test.tsx`, "sign-out failure path…" — a 500 from `/auth/signout` still fires `onSignedOut`). Judged **not a genuine AC-3 violation**: `LocalDevProvider.SignOutAsync` (`backend/src/Hap.Api/Identity/LocalDevProvider.cs:92-93`) is a bare `HttpContext.SignOutAsync` clearing a cookie — no failure mode in this codebase today returns non-2xx from it — and this exact tradeoff was already flagged by `hap-code-reviewer` as pre-existing HAP-4 behaviour, not introduced by this story. Carried forward as a robustness note, not a blocker.

4. **DESIGN.md A6 conformance: tokens only, strings externalised, colour never the sole signal.** — PASS. `grep -nE '#[0-9a-fA-F]{3,6}' AppShell.css` → zero matches (re-verified independently of the design reviewer's claim). `git diff` on `strings/en.ts` shows `rolePlaceholder` fully removed and replaced by `signedInAs`/`identityUnavailable`/`individualRoleLabel`/`roleLabelPlatformAdmin`/`roleLabelHigExecutive`/`roleLabelBuDelegate`/`roleLabelGroupViewer`/`signOut` — no literal copy left in the component. 5. **vitest-axe passes on the shell with the new control; keyboard-only sign-out works.** — PASS on axe (re-ran; zero violations, 10/10 shell tests green). **Keyboard-only claim — probed and narrowed:** the pre-existing "sign-out works from the keyboard alone" test fires *both* a synthetic keydown *and* a synthetic click, so it cannot distinguish "Enter triggers it" from "the click handler works" — it would pass even if Enter did nothing. Isolated the keydown in a new test (`shell.test.tsx`, "a keydown alone (no click) does not trigger sign-out…"): confirmed that under jsdom, a keydown alone does **not** invoke `onSignedOut` — this is a jsdom limitation (it doesn't implement a real browser's native conversion of Enter-on-a-focused-`<button>` into a click), not a component defect. `@testing-library/user-event` (which polyfills that behaviour) is not a declared or installed dependency in this project — confirmed by `ls node_modules/@testing-library` — and adding one mid-QA would trip the "any new dependency" L2 trigger the existing L1 panel never reviewed, so I did not add it. What the new test proves instead, and what real keyboard operability actually rests on: the sign-out element is an unadorned native `<button type="button">` with no `onkeydown` handler of its own to intercept or mis-implement the browser's native Enter/Space activation — verified directly (`signOutButton.onkeydown` is `null`, no `onkeydown` attribute). Verdict: keyboard operability holds structurally; the existing test's title overclaims what it proves, which is a test-quality note, not an AC violation.
